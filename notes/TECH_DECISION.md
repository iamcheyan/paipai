# 技术选型：为什么做 App 不做 Web PWA

**决策**: Expo (React Native)
**日期**: 2026-06-26
**状态**: 已定

---

## 问题

Web PWA 开发更快，但团队担心演示效果，认为 App 更符合产品设想。

## 结论

**做 App（Expo），不做 Web PWA。** 团队的产品直觉是对的。

---

## 理由

### 1. 产品形态决定演示形态

拍拍的核心画面是 **全屏相机 + 半透明叠图 + 实时提示**。这是典型的原生手机体验：

- 用户预期：打开 App → 举起手机 → 对齐拍
- Web 实际：浏览器地址栏、权限弹窗、Safari 相机限制、很难「沉浸」

评委看 30 秒 demo，**像不像真的摄影 App** 直接影响「你们懂不懂这个产品」的印象。

### 2. Web 相机的硬伤（尤其 iOS）

| 问题 | Web PWA | Expo |
|------|---------|------|
| 全屏沉浸 | 难，有浏览器 UI | ✅ 原生全屏 |
| 相机权限 | Safari 限制多 | ✅ expo-camera 成熟 |
| 陀螺仪/方向 | API 有，体验参差 | ✅ 可用，demo 可 mock |
| 「这是个 App」的感觉 | ❌ 像网页 | ✅ 像产品 |
| 地图 + 相机切换 | 两个 WebView 感 | ✅ 原生导航 |

### 3. 开发速度差距没那么大（有 Expo + AI）

Web 快 2–3 小时，但省下的时间换不来演示说服力。

你们配置：
- 2 开发者 + AI 协调
- 已开 Maps SDK for Android/iOS + Places API（正好配 Expo）
- 社交/榜单全 mock → 后端零工作量

**Expo 一天交付范围**（ realistic）：
- 3 个页面：地图 → 地点榜单 → 相机跟拍
- 1 个点位真 overlay
- 点赞本地 +1

这与 Web PWA 能交付的功能 **一样多**，只是包装成 App。

### 4. 已开的 API 也指向 App

你们开了 **Maps SDK for Android/iOS**，没开 Maps JavaScript API——说明 Cloud 侧已经按 App 路径走了。改 Web 还要再开/再配一套。

---

## 如何降低 App 路径的风险

### 用 Expo Go，不折腾打包

明天 demo：
1. 开发机跑 `npx expo start`
2. 全员手机装 **Expo Go**
3. 扫 QR code → 真机演示

不需要 App Store、不需要 EAS Build（除非现场网络极差，可今晚打一个 dev build 备用）。

### 砍 scope 保 App 核心体验

| 保（App 价值） | 砍（省时间） |
|----------------|--------------|
| 全屏相机 + 叠图 | 多点位机位库 |
| 地图 + 1 个 POI | Places 实时搜索 |
| 榜单 UI | 真后端 |
| 拍照 → 本地榜单刷新 | 登录、留言持久化 |

### Backup 策略

| 风险 | 应对 |
|------|------|
| 现场 Wi-Fi 差，Expo Go 连不上 | 今晚录 30s 跟拍视频 `demo/videos/` |
| 相机 bug | 设计师预拍 before/after 对比图 |
| 地图 API 挂 | 静态地图截图 + 手动进地点页 |

## 一句话给评委

> 「拍拍是手机上的取景教练——所以我们用真机 demo，不是网页。」

这本身也是产品叙事的一部分。
